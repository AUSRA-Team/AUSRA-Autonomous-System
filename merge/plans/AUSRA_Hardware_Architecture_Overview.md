# AUSRA Hardware Architecture Overview

This document provides a comprehensive overview of the physical hardware software architecture for the AUSRA autonomous swarm robots, based on the `hardware_full_stack.launch.py` configuration.

## 1. High-Level Launch Sequence

The launch file employs a timed sequence to ensure dependencies are met before subsequent components are initialized. 

- **Stage 0: Core Hardware & Description (Immediate)**
  Initializes the fundamental robot state, URDF description, and physical hardware drivers (motors and LiDAR).
- **Stage 1: Localization & SLAM (5-second delay)**
  Starts the Extended Kalman Filter (EKF) and asynchronous SLAM Toolbox to begin tracking the robot's pose and building the map based on the active hardware sensors.
- **Stage 2: Nav2 Navigation (15-second delay)**
  Boots the Nav2 stack, allowing time for the SLAM node to establish the `map` TF frame.
- **Stage 3: Exploration (30-second delay)**
  Launches the frontier exploration server (`explore_lite`) to automatically drive the robot to unknown map regions, giving Nav2 enough time to become fully active.
- **Stage 4: Optional Nudge (35-second delay)**
  If the `nudge_robot` argument is true, it publishes a brief velocity command (`/cmd_vel`) to slightly rotate the robot, helping to seed the SLAM algorithm.

## 2. Core Nodes and Namespaces

There are no explicit namespaces applied at the top level of this launch file, meaning nodes operate in the global namespace by default.

### Core Nodes:
- **`robot_state_publisher`**: Parses the robot's URDF (`robot.urdf.xacro`) and publishes static transforms.
- **`omnidirectional_driver` (`omni_driver`)**: The custom hardware driver managing the holonomic base. Uses parameters from `hardware_params.yaml`.
- **LiDAR Driver**: Uses `sllidar_ros2` (`sllidar_a1_launch.py`) to interface with the physical LiDAR sensor.
- **Localization**: Uses `ausra_localization` (`localization.launch.py`) to run the EKF for fused odometry.
- **SLAM Toolbox**: Runs `async_slam_toolbox_node` in mapping mode, configured via `slam_toolbox_config.yaml`.
- **Nav2**: Uses the standard `nav2_bringup` (`navigation_launch.py`) configured with holonomic parameters derived from simulation (`nav2_holonomic_params.yaml`).
- **Exploration**: Runs `explore_lite` (`explore_node`) using `explore_params.yaml`.

## 3. Core Topics and TF Tree Architecture

### Core Topics
- **`/scan`**: Laser scan data published by the `sllidar_ros2` driver, subscribed to by SLAM Toolbox and Nav2.
- **`/cmd_vel`**: Velocity commands generated by Nav2 and subscribed to by the `omnidirectional_driver`.

### TF Tree Flow
Based on the frame IDs specified in the launch parameters, the transform tree is structured as follows:

`map`  ➡  `ausrabot_odom`  ➡  `ausrabot_robot_footprint`  ➡  *(Fixed URDF Links)*  ➡  `ausrabot_lidar`

- **`map`**: Global fixed frame, published by SLAM Toolbox.
- **`ausrabot_odom`**: The odometry frame, typically published by the EKF node.
- **`ausrabot_robot_footprint`**: The base frame of the robot, centered on the ground.
- **`ausrabot_lidar`**: The sensor frame, set by the LiDAR driver and statically linked in the URDF.

## 4. Hardware Interfaces and Drivers

The launch file identifies two primary physical hardware interfaces:

1. **RPLidar A1 Sensor**: 
   - Uses the `sllidar_a1_launch.py` script from the `sllidar_ros2` package.
   - Communicates over serial port: `/dev/ttyUSB0`.
   - Baudrate: `115200`.
   
2. **Omnidirectional Base**:
   - Custom `omni_driver` from the `omnidirectional_driver` package.
   - Responsible for converting `/cmd_vel` twists into physical motor commands for a holonomic drive system.
